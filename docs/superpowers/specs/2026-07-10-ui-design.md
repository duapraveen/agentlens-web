# AgentLens UI Design

Date: 2026-07-10  
Status: Approved  
Spec: 001-agentlens-core

---

## Overview

A multi-page Streamlit app (`agentlens/dashboard/`) exposing all platform functionality across three user personas. Role selection is a sidebar dropdown (session state only — no auth). Navigation items filter by the active role. All pages live as separate files under `agentlens/dashboard/pages/`.

---

## Role Model

Role is selected via `st.selectbox` at the top of the sidebar. The selection is stored in `st.session_state["role"]`. Pages not available to the active role are excluded from the sidebar nav.

| Page | Engineer | Reviewer | Lead |
|---|---|---|---|
| Overview | ✓ | ✓ | ✓ |
| Conversations | ✓ | | ✓ |
| Call Detail | ✓ (via Conversations) | ✓ (via Review Queue) | |
| Clusters | ✓ | | ✓ |
| Review Queue | | ✓ | |
| Fix Workbench | ✓ | | |
| Jobs | ✓ | | |

Call Detail is not a direct sidebar item. It is navigated to via `st.switch_page()` with the selected `call_id` stored in `st.session_state["selected_call_id"]`.

## Pages

### 1. Jobs

**Role:** Engineer only.  
**Purpose:** Seed the system. Trigger corpus generation, eval runs, and clustering. Nothing else in the app works without data from here.

**Layout:**

Three cards stacked vertically — Corpus Generation, Eval Run, Clustering — each with controls and a one-line summary of the last run. Below all three: a scrollable job log panel showing the last 20 lines from `logs/jobs.log`.

**Corpus Generation card:**
- `st.number_input` for call count (default 200)
- `st.slider` for failure injection rate (default 30%)
- `[ Generate Corpus ]` button
- Last-run summary: timestamp, call count, duration

**Eval Run card:**
- Radio: `Full corpus` / `Unevaluated only` (default: Unevaluated only)
- `st.selectbox` for judge model (default: `claude-haiku-4-5`)
- Estimated cost display (computed client-side from call count × per-call estimate)
- `[ Run Evals ]` button
- Last-run summary: timestamp, calls evaluated, actual cost, duration

**Clustering card:**
- `[ Re-cluster Failures ]` button
- Last-run summary: timestamp, cluster count, failure count

**Job Log panel:**
- Fixed-height scrollable container (`st.container(height=200)`)
- Streams stdout from `python -m agentlens.jobs.*` via `subprocess` + `st.empty()`
- Format: `[HH:MM] <job>  <status>  <summary>`

---

### 2. Conversations

**Role:** Engineer, Lead.  
**Purpose:** Browse all evaluated calls. Entry point for drilling into a specific call.

**Layout:**

Filter bar across the top, summary line, then a paginated `st.dataframe`.

**Filters:**
- `Severity`: multiselect — All / P0 / P1 / P2
- `Dimension`: selectbox — All / task_completion / factual_accuracy / safety_compliance / communication_quality
- `Cluster`: selectbox — All + cluster names from DB
- Toggle buttons: `Pass only` / `Fail only`

**Summary line:** `187 calls · 61 with failures · 3 P0`

**Table columns:** ID, Scenario, Fails (four-dot indicator), P0 flag, Avg Score, Cost, Date

**Dot indicator:** Four dots representing the four rubric dimensions in fixed order (task_completion, factual_accuracy, safety_compliance, communication_quality). Filled dot = failed dimension, empty = passed. Rendered as a string of unicode characters, e.g. `●●○○`.

**P0 flag:** `⚠` character in its own column. Rows with any P0 finding get a red left border via custom CSS.

**Pagination:** 25 rows per page, prev/next controls below the table.

**Row click:** Clicking a row sets `st.session_state["selected_call_id"]` and calls `st.switch_page("pages/call_detail.py")`.

---

### 3. Call Detail

**Role:** Engineer (via Conversations), Reviewer (via Review Queue).  
**Purpose:** Full diagnostic view of a single evaluated call.

**Navigation header:** `← Back` link that returns to whichever page the user came from (stored in `st.session_state["call_detail_origin"]`). Cluster name shown as a link that navigates to the Clusters page filtered to that cluster.

**Transcript panel:**
- Fixed-height scrollable container (`st.container(height=300)`)
- Alternating speaker labels: `Patient:` / `Agent:`

**Scores section:**
- Rendered as one `st.expander` per dimension (not a `st.dataframe` — expansion must be inline)
- Expander header: dimension name · score (0–100) · severity badge · pass/fail indicator · pipeline stage
- Expanding reveals the judge reasoning and deterministic check results for that dimension

**Judge reasoning expander (per dimension):**
- Judge's reasoning text
- Deterministic check results — always visible, not nested further. Format: `⚠ MISSED_ESCALATION_KEYWORD` or `✓ No deterministic flags`
- Metadata footer: `Prompt v1.2 · Model: haiku-4-5 · Rubric v1.0 · Input hash: abc123`

**Dev mode toggle:**
- `st.toggle("Show ground truth")` — hidden by default, visible to Engineer role only
- When on: shows injected failure mode, pipeline stage, and severity from ground-truth labels alongside judge output

---

### 4. Clusters

**Role:** Engineer, Lead.  
**Purpose:** View recurring failure patterns and initiate fixes.

**Layout:**

Filter bar, summary line, then a vertical stack of cluster cards.

**Filters:**
- `Routing`: selectbox — All / prompt_fix / retrieval_data_fix / ops_process / model_config
- `Severity`: selectbox — All / P0 / P1 / P2

**Summary line:** `14 clusters · 89 failures · last clustered 15:06`

**Cluster card:**
- Header: severity icon (⚠ for P0) + cluster label + call count
- Sub-line: routing suggestion + dominant severity
- Description: LLM-generated summary (one to two sentences)
- Actions: `[ View N calls ]` → Conversations page filtered to cluster. `[ Propose Fix ]` → Fix Workbench with cluster pre-selected (disabled on P0 clusters with tooltip: "P0 findings require human resolution before a fix can be proposed")
- P0 cards: red left border, sorted to top regardless of filter

---

### 5. Review Queue

**Role:** Reviewer only.  
**Purpose:** Human calibration of judge findings. One finding at a time.

**Layout:**

Agreement stats panel at the top (always visible), then the current finding card, then the verdict + note + submit controls.

**Agreement stats panel:**
- Overall agreement % + per-dimension breakdown
- Review count today + pending count in queue
- Updates via `st.rerun()` after each submission

**Finding card:**
- Call ID, scenario, dimension, score
- Judge finding text
- Deterministic check result (always visible)
- `[ View full transcript ]` — expands inline via `st.expander`; does not navigate away

**Verdict controls:**
- Two toggle buttons: `[ ✓ Agree ]` and `[ ✗ Disagree ]` — mutually exclusive selection, highlighted when chosen. Clicking a selected button deselects it. Implemented as `st.radio` styled as buttons, or `st.session_state` toggle with conditional `st.button` styling.
- Note field: `st.text_area("Note (optional)")` — always visible regardless of verdict selection
- `[ Submit & Next ]` — saves verdict + note to DB and advances to next finding. Disabled until a verdict is selected. When the queue is empty after submission, shows a "Queue clear" success state with final agreement stats.

**Behavior:** Clicking Agree or Disagree does NOT submit or navigate. Only "Submit & Next" advances the queue. This allows the reviewer to change their verdict, add or edit a note, and submit deliberately.

---

### 6. Overview

**Role:** All.  
**Purpose:** Aggregate quality health at a glance. Default landing page.

**Layout:** Four panels in a 2×2 grid using `st.columns`.

**Quality panel (top-left):**
- One row per dimension: label + `st.progress` bar (pass rate) + percentage + 7-day delta with ▲/▼ arrow
- Delta compares last 7 days vs prior 7 days

**Severity panel (top-right):**
- P0 / P1 / P2 counts with colored dot indicators
- Each count is a clickable link → Conversations filtered to that severity

**Judge Accuracy panel (bottom-left):**
- Precision, Recall (vs golden set ground truth)
- Human agreement % + review count
- All three metrics sourced from the same calibration stats used by the Review Queue

**Top Clusters panel (bottom-right):**
- Top 5 clusters by call count: label, count, severity, routing suggestion
- Each row is a link → Clusters page filtered to that cluster

**Cost panel (below the grid):**
- Total eval cost to date + average cost per call

---

### 7. Fix Workbench

**Role:** Engineer only.  
**Purpose:** Propose a fix for a failure cluster, run regression, review before/after delta.

**Layout:** Cluster selector at top, then Proposed Fix card, then Regression Results card (hidden until a regression has been run).

**Cluster selector:**
- `st.selectbox` listing all non-P0 clusters. If navigated from the Clusters page, pre-selected to that cluster via `st.session_state["fix_cluster_id"]`

**Proposed Fix card:**
- Fix type (prompt_fix / retrieval_data_fix / ops_process / model_config)
- Rationale text (LLM-generated)
- Patch content in a code block (`st.code`)
- `[ Generate Fix ]` — calls LLM, renders result. Idempotent; re-clicking regenerates.
- `[ Apply & Run Regression ]` — re-generates affected scenarios with patched agent, re-runs evals, populates the Regression Results card

**Regression Results card (shown after run):**
- Table: Dimension / Before / After / Delta / Status
- Delta formatting: positive in green (▲), negative in yellow (▼), zero as `—`
- A drop of any magnitude shows ⚠ in the Status column; a drop > 5pp adds an explicit warning banner above the table
- P0 cluster guard: if a P0 cluster is somehow selected (e.g., via session state), `[ Apply & Run Regression ]` is disabled with tooltip "P0 findings require human acknowledgment before regression can run"

## Navigation Flows

```
Jobs
  └─► (data exists) ──► Conversations
                              └─► Call Detail ──► Clusters
                                                      └─► Fix Workbench

Review Queue ──► Call Detail (read-only transcript)

Overview ──► Conversations (severity filter)
         └─► Clusters (cluster filter)
```

## Shared Conventions

- **Session state keys:** `role`, `selected_call_id`, `call_detail_origin`, `fix_cluster_id`
- **Custom CSS:** injected via `st.markdown("<style>...</style>", unsafe_allow_html=True)` in the app shell for row borders and button toggle styles
- **No raw SQL:** all DB access via SQLAlchemy 2.0 ORM
- **No transcript content in logs:** log call IDs and metadata only
- **All LLM calls** (fix generation, cluster labeling) go through `agentlens/llm/gateway.py`
